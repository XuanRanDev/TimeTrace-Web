# 命名规则

命名规则用于控制文件归档后的**文件名格式**。默认情况下，时迹归档文件时保留原始文件名不变；命名规则则允许您为满足特定条件的文件，指定一套统一的重命名模板。

这个功能主要解决以下痛点：

- 来自不同设备、不同应用的文件命名格式五花八门（`mmexport1698025260832.jpg`、`IMG_001.jpg`、`DCIM_0001.jpg`……），归档后依然混乱
- 哈希文件名（如 `a3f2b1c4d5e6f708.jpg`）完全无法从文件名判断内容，不便检索
- 希望所有归档文件都遵循统一的命名规范，便于长期管理

与分类规则一样，命名规则采用「条件 + 模板」的结构，按优先级从高到低匹配，首条命中即生效；不满足任何规则条件的文件保留原始文件名。每个用户拥有独立的命名规则集，各用户互不影响。

---

## 命名条件：什么样的文件应用这条规则

条件可以**留空**（表示对所有文件生效），也可以指定以下一种或多种，**多个条件用英文逗号分隔，满足其中任意一条即触发**：

**`prefix:值`**：文件名以指定字符串开头（不含扩展名）。例如 `prefix:mmexport` 可匹配微信导出的文件，`prefix:IMG_` 匹配相机默认命名。

**`suffix:值`**：文件名以指定字符串结尾（不含扩展名）。例如 `suffix:_edit` 可匹配手动编辑过的文件。

**`folder:值`**：文件的**直接来源文件夹**名称中包含指定字符串。例如 `folder:DCIM` 匹配来自相机存储卡目录的文件，`folder:Telegram` 匹配从 Telegram 下载的文件。

**`hash`**：文件名形似哈希值或随机字符串——即名称中没有明显的日期或有意义的词汇，看起来像是系统自动生成的随机标识符。时迹通过内置算法判断，这个条件无需您手动指定特征值。

**`regex:表达式`**：使用正则表达式匹配文件名（不含扩展名）。适合需要精确控制匹配逻辑的场景，例如 `regex:^\d{10,13}$` 可匹配纯数字文件名。

---

## 命名模板：定义目标文件名格式

模板是一个字符串，其中可以嵌入以下变量，归档时会自动替换为该文件对应的实际值：

| 变量 | 含义 | 示例值 |
|------|------|--------|
| `{date}` | 归档日期时间（紧凑格式） | `20240115_143022` |
| `{date_short}` | 归档日期（紧凑格式，不含时间） | `20240115` |
| `{year}` | 年份（4 位） | `2024` |
| `{month}` | 月份（2 位，补零） | `01` |
| `{day}` | 日（2 位，补零） | `15` |
| `{time}` | 时间（6 位紧凑格式） | `143022` |
| `{sha1}` | 文件 SHA1 哈希前 6 位 | `a3f2b1` |
| `{sha1_full}` | 文件完整 SHA1 哈希值 | `a3f2b1c4d5e6...` |
| `{ext}` | 文件扩展名（含点号） | `.jpg` |
| `{orig}` | 原始文件名（不含扩展名） | `IMG_001` |
| `{folder}` | 来源文件夹名称 | `Camera` |
| `{make}` | EXIF 相机品牌 | `Apple` |
| `{model}` | EXIF 相机型号 | `iPhone 15 Pro` |
| `{counter}` | 本批次归档中的自增序号（从 1 开始） | `1` |

模板中**必须包含 `{ext}`**，否则归档后的文件将没有扩展名，导致系统无法识别文件类型。

## 实用配置示例

**场景一：统一处理微信导出文件和哈希文件名**

很多用户从微信导出的媒体文件名形如 `mmexport1698025260832.jpg`，文件名是时间戳；另有一些文件名是完全随机的哈希字符串，毫无辨识度。可以用一条规则统一重命名：

- 条件：`prefix:mmexport,hash`
- 模板：`{date}_{sha1}{ext}`
- 效果：`mmexport1698025260832.jpg` → `20231023_143022_a3f2b1.jpg`

**场景二：为所有文件统一加上日期前缀，保留原始文件名**

如果您希望归档后的文件名既保留原始信息，又方便按文件名排序：

- 条件：（留空，对所有文件生效）
- 模板：`{date_short}_{orig}{ext}`
- 效果：`IMG_001.jpg` → `20240115_IMG_001.jpg`

**场景三：对某个特定来源目录的文件统一命名**

例如，来自 Telegram 下载目录的文件通常命名混乱，希望统一处理：

- 条件：`folder:Telegram`
- 模板：`{date}_{sha1}{ext}`
- 效果：`photo_2024-01@23-14-30-22.jpg` → `20240123_143022_a3f2b1.jpg`

## 优先级与匹配冲突

与分类规则一样，命名规则按优先级从高到低匹配，首条命中即生效。这意味着：

- 将条件更具体的规则设为更高优先级，避免被宽泛规则过早拦截
- 「留空条件（对所有文件生效）」的规则优先级应设为最低，否则它会让所有其他规则失效

---

## 命名规则与分类规则的关系

命名规则和分类规则相互独立，互不影响。同一个文件可以同时命中一条命名规则（决定文件名）和一条分类规则（决定所在子目录），两者分别从各自的规则集中匹配，结果合并应用。

---

## 生效时机

命名规则仅在**归档阶段**生效，修改规则不会对已完成归档的历史文件产生影响。若需按新命名规则处理历史文件，需要重新执行归档流程。已归档文件在文件浏览中的「格式化文件名」字段会展示时迹为其计算的命名结果，可在实际归档前用于核对。
