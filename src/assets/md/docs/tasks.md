# 任务管理

每一次扫描或归档操作在时迹内部都对应一条「任务记录」，任务管理页面是这些记录的统一视图。您可以在这里查看每次操作的执行结果和详细统计，了解时迹对您的文件究竟做了什么。

理解时迹的一个关键点是：**扫描和归档是两个相互独立的阶段**，分别对应两种任务类型。扫描任务负责「分析」——遍历目录、提取时间、计算哈希、生成文件记录；归档任务负责「执行」——将扫描阶段已分析好的待处理文件，按照既定路径复制或移动到目标目录。这两个阶段的分离设计，让您可以在扫描完成后先检视文件浏览中的记录，确认提取结果符合预期，再决定是否归档。

---

## 任务类型

**扫描任务**：由您手动新建或由定时调度触发。时迹对源目录进行递归遍历，对新增或变化的文件依次执行：增量判断 → SHA1 哈希计算 → 时间信息提取 → 分类与命名预计算，最终将结果写入文件记录（状态为「待处理」或「已跳过」）。扫描任务只分析，不移动任何文件。

**归档任务**：在扫描完成后自动触发（若任务开启了「扫描后自动归档」），或在文件浏览中手动勾选待处理记录后触发。归档任务读取这些「待处理」记录中已预计算好的目标路径，将文件复制或移动过去，并将记录状态更新为「已归档」或「错误」。

> 在您不确定某些文件是否能被正确识别并归档时，建议在新建任务时关闭自动归档，待扫描完成后核对其相关信息，若无误手动归档即可；若存在问题，可直接删除相关记录，自行进一步处理使得时迹能够正确识别。（在已扫描未归档时由于文件位置未发生变动，因此可以自由删改文件记录，但若文件已归档则不建议删除文件记录）

## 任务状态与并发控制

| 状态 | 含义 |
|------|------|
| 运行中 | 任务正在执行 |
| 已完成 | 任务正常结束，所有文件处理完毕 |
| 已中断 | 应用重启或异常退出导致任务中途停止 |
| 失败 | 任务遇到无法继续的错误，提前终止 |

时迹在全局层面实施并发控制：**同一时刻最多只允许一个扫描任务和一个归档任务同时运行**。若您新建任务时已有同类任务在执行，新任务会等待当前任务完成后再开始，不会并行抢占资源或互相干扰。这一设计避免了多任务同时操作相同目录时可能产生的数据竞争问题。

若归档任务因应用重启等原因被中断，重新启动应用后，该任务会自动转为「已中断」状态，并在任务列表中显示「重试」按钮。点击「重试」后，时迹会从已完成的文件继续向后处理，无需从头开始。**扫描任务不支持断点续传**，若中断则需重新新建扫描任务——由于增量机制的存在，已处理过的文件会被自动跳过，重新扫描不会产生重复文件。

## 新建扫描任务

点击页面右上角的**新建扫描任务**按钮，填写以下配置：

**源目录**：时迹将从这个目录开始递归扫描所有子目录和文件。目录须在 fnOS 应用设置中已授权给时迹，才会出现在选择器中。若未看到您期望的目录，请联系管理员在应用设置中完成目录授权。

**目标目录**：归档后的文件将按 `{年}/{月}/{分类}/` 结构存放在这个目录下。目标目录同样须在授权范围内，且建议选择与源目录不同的位置，避免时迹将自己归档好的文件在下次扫描时再次处理。

**归档模式**：
- **复制**：保留源文件不动，在目标目录生成副本。适合初次使用、验证归档结果的场景，即使出现配置错误，源文件也不会丢失
- **移动**：将文件从源目录迁移到目标目录，源文件归档后删除。适合整理已确认无误、希望彻底迁移的大批量文件

> 强烈建议在首次使用时选择「复制」模式，熟悉时迹的归档逻辑并确认结果符合预期之后，再视情况切换为「移动」模式。

**扫描后自动归档**：开启时，扫描完成后时迹会立即自动触发归档；关闭时，扫描结果以「待处理」状态停留在文件浏览中，由您在检视完毕后手动触发归档。若您希望在归档前逐一确认文件的时间提取结果，建议关闭此选项。

## 读懂任务统计数据

点击任意任务行展开详情，可以看到以下统计字段。以扫描任务为例：

**新增**：本次扫描中首次出现的文件数量，或之前记录已删除、现在重新出现的文件。

**变化**：与上次扫描相比，文件大小或修改时间发生了变化，时迹重新对其进行了时间提取和哈希计算。

**未变化**：大小与修改时间均与上次一致，直接跳过，不重新处理。扫描速度越来越快，通常正是因为这类文件的比例在增高。

**消失**：上次扫描时存在、本次不存在的文件，通常意味着源文件已在扫描期间被删除或移动。时迹会将其记录为「消失」，但不会主动对目标目录中的已归档副本做任何操作。

**成功 / 跳过 / 出错**：归档任务专有统计。成功代表文件已成功复制或移动；跳过通常是因为目标路径已存在相同文件；出错的记录可在文件浏览中按「错误」状态筛选查看详细原因。

## 时间提取测试

工具栏中的**时间提取测试**功能，让您无需上传真实文件，就能验证时迹的时间提取逻辑对特定文件名的处理结果。这在以下场景中非常实用：

- 在开始大批量扫描前，用一批样本文件名预先验证提取结果是否符合预期
- 对某个提取结果异常的文件，分析时迹内部的判断依据
- 测试某类特殊命名格式（如设备导出的特殊文件名）是否能被正确识别

**使用方法**：在弹窗的文本框中输入文件名，每行一个，最多 200 条，点击「测试」即可。

**结果说明**：

- 绿色行：成功提取到日期，同时显示采用的匹配模式（如 `prefix_datetime`、`unix_ms` 等）
- 红色行：未能从文件名中提取到有效日期，实际归档时将以文件修改时间兜底

**展开候选详情**：点击任意结果行可以展开，查看时迹为该文件名找到的**所有候选日期**及各自的置信度分值。时迹的时间提取机制并非简单的「找到第一个匹配就停」，而是先收集所有可能的候选，为每个候选按上下文（前缀特征、数字段数量、日期是否在合理范围等）打分，最终选出置信度最高的那个。

展开候选列表后，您可以清楚地看到：时迹识别出了哪些候选、每个候选的分值是多少、以及最终为何选中了某个特定的日期。如果您认为选中的结果有偏差，候选列表可以帮助您判断问题所在——是提取本身的局限，还是文件名格式与预期不符。
